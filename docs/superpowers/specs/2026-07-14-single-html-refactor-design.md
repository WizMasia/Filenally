# File-nally 단일 HTML 전면 리팩터링 설계

## 1. 목표와 보존 조건

File-nally의 런타임 구현은 계속 `file-nally.html` 하나에만 존재한다. 사용자는 파일을 Chrome 또는 Edge로 열기만 하면 되며 Node.js, 번들러, 서버, 외부 CDN이 필요하지 않다.

다음 계약은 유지한다.

- 단일 페이지·단일 HTML 실행 파일
- File System Access API를 이용한 로컬 폴더 동기화
- 설정, 이력, 매니페스트의 JSON 저장과 백업·복원
- 한국어와 영어 UI
- 양방향 및 원본→대상 단방향 동기화
- 삭제 파일의 `.trash` 격리
- 비교 후 명시적으로 실행하는 안전 흐름

테스트, 설계 문서, 개발용 패키지 설정은 별도 파일로 둘 수 있지만 제품 구현 코드는 `file-nally.html` 밖으로 분리하지 않는다.

## 2. 단일 파일 내부 구조

`file-nally.html`은 물리적으로 하나의 파일을 유지하되 다음 순서의 논리 계층으로 재구성한다.

1. 디자인 토큰과 반응형 CSS
2. 정적 HTML과 인라인 SVG 아이콘 정의
3. 번역 사전
4. 공용 유틸리티와 입력 검증
5. JSON 상태 스키마·마이그레이션·저장소
6. 폴더 쌍 식별과 브라우저 핸들 저장소
7. 순수 동기화 계획 엔진
8. File System Access API 어댑터와 실행기
9. 안전한 DOM 렌더러
10. 애플리케이션 컨트롤러와 이벤트 연결

각 계층은 하나의 IIFE 또는 동결된 네임스페이스로 노출하며 전역 가변 상태를 하나의 `AppController`로 제한한다. 동기화 계획 엔진은 DOM과 파일 핸들에 의존하지 않는 순수 함수로 작성한다.

## 3. JSON 상태 스키마와 호환성

저장 키 `smart_sync_state`와 JSON 백업·복원 기능은 유지한다. 신규 상태는 `schemaVersion: 2`를 사용한다.

```json
{
  "schemaVersion": 2,
  "config": {
    "direction": "bidirectional",
    "conflictPolicy": "latest",
    "excludeDirs": ["node_modules", ".git", "dist", "temp", ".trash"],
    "lang": "ko"
  },
  "activeProfileId": null,
  "profiles": {},
  "globalHistory": []
}
```

각 프로필은 한 폴더 쌍에만 대응한다.

```json
{
  "id": "generated-id",
  "sourceName": "source-folder",
  "targetName": "target-folder",
  "bindingStatus": "verified",
  "createdAt": "ISO-8601",
  "lastUsedAt": "ISO-8601",
  "manifest": {},
  "history": []
}
```

기존 v0.6.1 JSON의 `config`, `history`, `manifest`는 자동으로 v2로 변환한다. 기존 매니페스트는 실제 폴더 쌍이 증명될 때까지 `unverified` 프로필로 취급하며 삭제 전파에 사용하지 않는다. 손상되거나 부분적인 JSON은 기본값과 병합하되 유효하지 않은 필드는 폐기하고 사용자에게 복원 결과를 알린다.

기존 `conflictPolicy: "overwrite"`는 `source-overwrite`로 변환한다. 가져오기는 5MB, 프로필 100개, 프로필당 매니페스트 100,000개, 프로필당 이력 30개, 전체 이력 100개를 상한으로 한다. `__proto__`, `prototype`, `constructor` 키가 어느 깊이에서든 발견되면 가져오기를 거부한다.

JSON 내보내기에는 스키마 버전, 설정, 프로필별 매니페스트, 이력이 모두 포함된다. 브라우저 권한 토큰과 파일 핸들은 직렬화하거나 내보내지 않는다.

## 4. 폴더 쌍 식별과 권한

설정·이력·매니페스트는 계속 JSON으로 저장한다. 직렬화할 수 없는 `FileSystemDirectoryHandle`만 IndexedDB에 보조 저장하여 프로필과 실제 폴더 쌍을 연결한다.

- 선택한 폴더는 기존 핸들과 `isSameEntry()`로 비교한다.
- 양쪽 핸들이 모두 일치할 때만 기존 프로필의 매니페스트를 신뢰한다.
- 일치하는 프로필이 없으면 빈 매니페스트의 새 프로필을 만든다.
- IndexedDB를 사용할 수 없으면 현재 세션에서만 검증된 프로필로 동작하고, 새 세션에서는 삭제 전파를 비활성화한다.
- 같은 폴더를 양쪽에 선택하면 거부한다.
- `resolve()`가 지원되면 한 폴더가 다른 폴더의 하위인 경우도 거부한다.
- 새로고침 후 저장된 핸들을 복구할 때는 `queryPermission()`으로 확인하고, 사용자 동작이 있을 때만 `requestPermission()`을 호출한다.

## 5. 동기화 계획 규칙

비교 단계는 파일을 변경하거나 매니페스트를 수정하지 않는다. 스캔 결과와 신뢰된 이전 매니페스트를 입력으로 받아 실행 계획만 만든다.

파일 상태는 `path`, `size`, `lastModified`로 비교한다. 매니페스트의 이전 상태와 시간 및 크기가 모두 같을 때만 변경 없음으로 본다.

### 방향

- `bidirectional`: 양쪽의 신규·수정·삭제를 분석한다.
- `unidirectional`: 원본이 권위 있는 쪽이다. 대상의 단독 변경은 원본을 덮어쓰지 않는다.

### 충돌 정책

- `latest`: 한쪽만 변경됐거나 수정 시간이 다르면 최신 파일이 이긴다. 양쪽이 모두 변경되고 시간이 같은데 내용 크기가 다르면 자동 덮어쓰지 않고 충돌로 남긴다.
- `source-overwrite`: 양쪽 변경 충돌에서 원본이 이긴다. UI 라벨도 “원본 우선 덮어쓰기”로 명확히 바꾼다.
- `skip`: 대상 경로에 파일이 이미 있거나 양쪽 변경 충돌이면 아무 파일도 덮어쓰지 않는다.
- `rename`: 충돌한 양쪽 버전을 모두 보존한다. 각 폴더에 반대쪽 파일을 `<이름>.conflict-<source|target>-<시각>.<확장자>`로 복사하여 결과적으로 양쪽 폴더에 두 버전이 존재하게 한다.

### 삭제

- 삭제 전파는 검증된 폴더 쌍의 매니페스트가 있을 때만 허용한다.
- 한쪽 파일이 사라지고 반대쪽 파일이 이전 매니페스트 이후 수정되지 않았을 때만 반대쪽 파일을 휴지통으로 이동한다.
- 남아 있는 파일도 변경됐다면 삭제로 단정하지 않고 충돌 상태로 남긴다.
- 양쪽 모두 사라진 항목은 동기화 성공 시 매니페스트에서 제거한다.

## 6. 파일 작업 안전성

- 파일 쓰기는 `createWritable()` 완료 후에만 성공으로 기록한다.
- `.trash/<실행시각>/<원래 상대 경로>`에 격리하여 이전 휴지통 파일을 덮어쓰지 않는다.
- 각 파일 작업 성공 직후 메모리 매니페스트를 갱신하고, 체크포인트 단위로 JSON 상태를 저장한다.
- 파일 복사, 쓰기, 삭제 또는 휴지통 이동 중 하나라도 실패하면 해당 항목을 오류로 기록하고 전체 작업을 중단한다. 이미 성공한 항목의 매니페스트 체크포인트는 보존하며 사용자가 다시 비교해 재개할 수 있게 한다.
- 중단 요청은 현재 파일의 쓰기 또는 휴지통 이동이 끝난 뒤 다음 항목 전에 적용한다.
- 폴더 선택이나 설정이 바뀌면 기존 실행 계획을 즉시 폐기한다. 실행 중에는 폴더와 설정 컨트롤을 잠그며, 실행은 마지막 비교에서 발급한 현재 계획 ID와 일치할 때만 시작한다.

## 7. 보안과 입력 검증

- 파일명, 상대 경로, 오류 메시지, JSON 이력 등 외부 데이터는 `innerHTML`에 삽입하지 않는다.
- 동적 UI는 `createElement()`와 `textContent`로 생성한다.
- JSON 가져오기는 허용된 키와 타입만 복사한다.
- 프로토타입 오염 키와 과도한 이력·매니페스트 크기를 거부한다.
- 파일 경로는 빈 구간, `.` 및 `..` 구간을 거부한다.
- 외부 스크립트와 외부 폰트를 사용하지 않는다.

## 8. UI와 접근성

현재의 밝은 운영 도구 성격과 인디고 중심 색상은 유지하되 단일 디자인 토큰 체계로 정리한다.

- `header`, `main`, `section`, `footer` 구조를 사용한다.
- 모든 아이콘은 HTML 내부 SVG 심볼로 제공하고 이모지를 UI 아이콘으로 사용하지 않는다.
- 760px 이하에서는 비교 카드와 이력 영역을 1열로 전환한다.
- 표는 최소 너비와 수평 스크롤을 사용하여 한글이 글자 단위로 세로 줄바꿈되지 않게 한다.
- 작업 버튼은 모바일에서 줄바꿈하고 화면 밖으로 넘치지 않는다.
- 모든 입력과 버튼에 키보드 포커스 표시를 제공한다.
- 상태 및 진행률은 `aria-live`와 `role="status"`로 전달한다.
- 일반 텍스트는 WCAG AA 4.5:1 이상의 대비를 사용한다.
- 위험 작업은 색상만이 아니라 텍스트와 아이콘으로도 구분한다.

## 9. 테스트 전략

제품 코드는 하나의 HTML에 유지하되 테스트는 별도 개발 파일로 둔다. 런타임 의존성은 추가하지 않는다.

### 순수 계획 엔진 테스트

- 양쪽 없음, 한쪽 신규, 양쪽 동일
- 원본 최신, 대상 최신
- 단방향에서 대상 변경 보호
- 각 충돌 정책
- 검증된 삭제, 미검증 삭제, 삭제와 수정 충돌
- 동일 수정 시간·다른 크기
- 제외 폴더와 중첩 경로

### Chrome 통합 테스트

- 폴더 선택 → 비교 → 실행 → 재비교
- 비교 완료 후 버튼 상태 복구
- 복사, 이름 변경 보존, `.trash` 격리
- 중단과 부분 성공 체크포인트
- JSON v1 마이그레이션, v2 내보내기·가져오기, 손상 JSON 거부
- 언어 전환과 새로고침 유지
- 악성 파일명과 악성 JSON 문자열이 텍스트로만 렌더링됨
- 같은 폴더 및 중첩 폴더 거부

### 시각·접근성 검증

- Chrome 375px, 768px, 1280px에서 가로 페이지 오버플로 없음
- 주요 상태의 전체 페이지 캡처
- 포커스, hover, disabled, 진행, 오류 상태 확인
- 색상 대비와 접근 가능한 이름 검사

## 10. 완료 기준

- 제품 구현은 `file-nally.html` 하나에만 존재한다.
- 기존 사용자 JSON이 데이터 손실 없이 마이그레이션된다.
- 폴더 쌍이 증명되지 않은 매니페스트로 삭제 작업을 만들지 않는다.
- 네 충돌 정책이 문서화된 동작대로 테스트된다.
- 외부 데이터가 HTML 또는 스크립트로 실행되지 않는다.
- 정상 UI 흐름에서 비교 후 동기화를 실행할 수 있다.
- 375px부터 1280px까지 레이아웃 오버플로가 없다.
- 자동 테스트, JavaScript 문법 검사, 실제 Chrome 기능 검증이 모두 통과한다.

## 11. 제외 범위

- 브라우저가 지원하지 않는 실제 절대 경로 표시
- 서버 동기화, 클라우드 저장소, 계정 시스템
- 파일 내용 해시를 위한 전체 파일 재읽기
- 백그라운드 자동 동기화와 OS 감시 서비스
- 다른 파일로 분리된 제품 JavaScript 또는 CSS
